# Phase 4: Data Parser & Validation Layer

**Goal:** Ensure that the data extracted from the pages is rigorously cleaned, normalized, and validated against strict human-body logic before it reaches the database.

> **Learning Objective:** Defensive data engineering, normalization logic, boundary checks, and confidence scoring models.

## What Gets Built
A robust intermediate layer where raw scraped text is interpreted based on category context. Malformed or abnormal data generated by UI variations will be flagged for manual review to preserve database integrity.

## Deliverables

### 1. Enhanced `parser.py`
Upgraded from Phase 2.
- Handles unit normalization (cm/inches).
- Region localization mapping.
- Maps varying category conventions into standardized types (e.g. `['t-shirt', 'polo', 'shirt', 'sweatshirt', 'hoodie', 'jacket', 'blazer']`).

### 2. `validator.py`
A new rules-based validation engine.
- Contains structural rules: checking min/max ranges for given human chest/shoulder boundaries.
- Context rules: enforcing that `body_hip` and `body_inseam` are null when the category implies top-wear.
- Confidence scoring: generates a `confidence` rating (`high`, `medium`, `low`) based on proximity to standard bounds or missing data markers.

### Validation Flow Architecture
```text
Raw Extracted Data -> Parser (Normalization) -> Validator -> [PASS, WARN, FAIL]
```
- **PASS/WARN:** Eligible for normal DB insertion.
- **FAIL:** Routed to a separate queue (the conceptually planned `flagged_entries`) for manual checking.

## Acceptance Criteria
- [ ] Impossible measurements (e.g., chest 250cm) result in a FAIL and are routed out.
- [ ] Bottom-wear fields attached to top-wear correctly trigger validation logic.
- [ ] Ambiguous sizes receive a lower confidence score but pass through validation.
